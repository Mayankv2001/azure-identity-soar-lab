# Telemetry Maturity Model

Part of the [Production Readiness & Operations Layer](../). This document
describes the levels a detection programme moves through as its telemetry matures
from **synthetic fixtures** to **full real ingestion**, what each level actually
buys you, and — honestly — where this repository sits.

> **Where this repo sits: Level 0 (Synthetic).** Every event in this lab is
> deterministically generated by [`src/generate_logs.py`](../../src/generate_logs.py)
> (seed 42, 7-day window, documentation IP ranges, `@contoso.com` personas). No
> data connector is configured, no Log Analytics workspace is receiving logs, and
> no detection has ever run against a live tenant. The maturity model below is the
> map I would follow to move a detection off synthetic data and into production —
> it is not a claim that any of that has happened here.

## Why a maturity model at all

A detection rule that passes its tests on synthetic data is not "done". Synthetic
telemetry is shaped for clarity: the attack is legible, the fields are always
present, the volume is trivial, and the thresholds are hand-tuned to the fixture.
Real telemetry is noisy, incomplete, high-volume and adversarial. The gap between
"fires correctly on the fixture" and "earns its alert volume in production" is
exactly what these levels measure.

Treating this as a graded progression does three things:

1. It stops a rule being called "production ready" when it has only ever seen
   synthetic data — the honest boundary this whole project insists on.
2. It makes the *next* investment obvious for each detection (get a real sample,
   measure false positives, baseline the threshold, cost the ingestion).
3. It ties telemetry maturity to the
   [Detection Quality Scorecard](../../security-engineering/detection-quality-scorecard.md):
   the scorecard measures **detection-as-code metadata quality** (is the rule
   mapped, tested, owned); this model measures **the data underneath it**. A rule
   can score 96/100 on metadata and still be Level 0 on telemetry. Both are true
   at once in this repo.

## The levels

| Level | Name | Data reality | What it proves | What it does *not* prove |
|-------|------|--------------|----------------|--------------------------|
| 0 | Synthetic | Deterministic generated fixtures; no connector | Logic, MITRE mapping, entity extraction, correlation, tests | Real precision, ingestion cost, field availability, timing |
| 1 | Sampled real | A small, point-in-time export of real logs (e.g. a KQL query result saved as CSV/JSON) replayed offline | Field names/shapes match the real schema; the rule survives real-world column drift | Sustained false-positive rate, volume/cost, coverage gaps |
| 2 | Partial real | Live connector for *some* sources; the rule runs scheduled in a test/dev workspace against a subset | Real precision on the connected sources, real latency, connector health | Full-estate coverage; cross-source correlation completeness |
| 3 | Full real | All required connectors live in production; rule scheduled, tuned, cost-reviewed, and change-controlled | Production readiness: measured precision, baselined thresholds, owned SLA, ingestion cost accepted | (nothing further — this is the target state) |

### Level 0 — Synthetic (current state)

**Data reality.** Fixtures only. `SigninLogs`, `AuditLogs`, `AzureActivity`,
`SecurityAlert` and `CyberArk_EPV_CL` are *modelled* — the KQL is written against
those real Sentinel table names and column shapes, but the rows come from the
generator, and watchlists (`CAPolicyAdmins`, `HighPrivilegeApps`,
`PrivilegedIdentities`, `AssetInventory`, and the control-plane equivalents) are
JSON reference files, not live `_GetWatchlist()` lookups.

**What it buys.** Everything you can prove *without* real data:

- Detection **logic** is correct and deterministic — the Python mirror in
  [`src/detection_engine.py`](../../src/detection_engine.py) asserts the same
  thresholds as the KQL, and CI fails on drift.
- **MITRE ATT&CK** mapping, **entity extraction**, **severity scoring** and
  **incident correlation** all work end to end (12 alerts → 8 incidents in the
  showcase run).
- The rule is **owned, versioned, documented and tested** — the metadata quality
  the scorecard rewards.

**What it deliberately does not buy.** Real precision, real ingestion cost, real
field availability, real timing/latency, or any evidence of behaviour under
production noise. Those require Level 1+.

### Level 1 — Sampled real

**Data reality.** A human runs the rule's query (or an equivalent) against a real
workspace *once*, exports a bounded sample (a few hours or days of the relevant
table), scrubs it, and replays it offline through the same engine.

**What it buys.** The first contact with reality — and it is cheap. You discover
whether the columns you assumed exist actually exist and are populated
(`SigninLogs.ResultType`, `AuditLogs.TargetResources`, `AzureActivity.OperationNameValue`),
whether your parsing survives real casing/nesting, and whether the benign
look-alikes documented in the
[Purple-Team Validation Pack](../../security-engineering/purple-team-validation.md)
actually show up. Most "the field was named differently in our tenant" surprises
are caught here, offline, before anything is scheduled.

**What it still does not buy.** A *sustained* false-positive rate (one sample is
a snapshot), volume/cost at estate scale, or coverage completeness.

### Level 2 — Partial real

**Data reality.** One or more data connectors are live in a **test/dev**
workspace, and the rule runs on a schedule against a subset of the estate (one
region, one business unit, or one source such as `SigninLogs` while `AuditLogs`
is still sampled).

**What it buys.** Real precision on the connected sources over time, real
scheduling latency, and connector *health* signals (heartbeat gaps, ingestion
delay, schema updates pushed by Microsoft). This is where you baseline the
threshold against real volume and where the tuning exclusions in the detection
YAML get validated against real benign traffic rather than the seeded
false-positive class.

**What it still does not buy.** Full-estate coverage, and — importantly for the
correlated rules (DET / CP-DET chains, `CP-DET-008`) — complete cross-source
correlation. A chain rule is only as good as its *least*-connected input, so a
partially-connected estate can silently miss chains.

### Level 3 — Full real

**Data reality.** Every required source for a given detection is live in
**production**: connectors healthy, watchlists populated from the real identity
and asset systems, the rule scheduled, tuned, cost-reviewed and under change
control.

**What it buys.** Production readiness in the honest sense the scorecard reserves
that phrase for — a measured precision, a baselined threshold, a named SLA, an
accepted ingestion cost, and an approval on record. Only at this level does a
detection graduate from "strong candidate (requires tenant-specific tuning)" to
genuinely deployed.

All **15** detections
([7 identity + 8 control-plane](../../security-engineering/detection-quality-scorecard.md),

the telemetry level are independent axes — the table makes that explicit:

## Exit criteria — how a detection *earns* the next level

A detection does not "graduate" by assertion. These are the gates I would hold it
to:

- **0 → 1:** a real, scrubbed sample of every required source has been replayed
  through the engine, and every required field was present and parsed. No new
  code paths silently `null`.
- **1 → 2:** the rule has run scheduled in a dev workspace for a defined baseline
  window; false positives are counted, dispositioned, and either tuned out with a
  versioned exclusion or accepted with rationale.
- **2 → 3:** all required connectors are live in production; ingestion cost is
  estimated and accepted by an owner; the threshold is baselined to real volume;
  an SLA and owner are recorded; and a change-control approval is on file.

This progression is intentionally slow. The failure mode a maturity model exists
to prevent is a synthetic-only rule being switched on in production and either
drowning the queue in false positives or — worse — sitting silent because a field
it assumed was never actually ingested.
